# P2-T1: Add FastAPI Endpoints

## Context
Phase 2 (API Layer), Task 1. Wraps the Phase 1 core pipeline in a REST API so it can be triggered remotely, integrated into workflows, and consumed by the frontend (P3) and scheduler (P5).

## 1. Add dependencies to `requirements.txt` (small)
- Add `fastapi>=0.115` and `uvicorn>=0.34`
- These are the only new dependencies needed — `httpx` is already present for `TestClient`

## 2. Create `src/api/` module (medium)

**Files:** `src/api/__init__.py`, `src/api/app.py`

**In-memory store** (module-level in `app.py`):
```python
_runs: list[PipelineRun] = []
_variants: list[ContentVariant] = []
```
Simple lists — no persistence yet (that's P6-T3). Cleared on server restart.

### Endpoints

#### `GET /news`
- Calls `fetch_news()` from `src/modules/ingestion`
- Returns `list[NewsItem]` directly (Pydantic models serialize automatically)
- No query params for MVP — returns whatever CoinGecko gives us
- **Why:** Lets the frontend (P3) and external consumers preview what news the agent is seeing without triggering a full pipeline run.

#### `POST /run`
- Calls `run_pipeline(trigger="api")` from `src/pipeline`
- Stores the returned `PipelineRun` in `_runs` and `ContentVariant` list in `_variants`
- Response schema:
  ```json
  {
    "run": { "PipelineRun fields" },
    "top_variants": [ { "ContentVariant fields" } ]
  }
  ```
- Returns `200` on success (even if pipeline status is `"completed"` with 0 articles)
- Returns `500` if `run_pipeline` raises an unhandled exception
- **Why:** Core endpoint that turns the CLI-only pipeline into an on-demand service — the scheduler (P5) and dashboard (P3) will both call this.

#### `POST /post`
- Request body: `{ "variant_id": "uuid-string" }`
- Looks up variant in `_variants` by ID
- If not found → `404` with error message
- If found → returns a `DistributionRecord` with `status="pending"` (stub until P4 wires up Twitter)
- **Why:** Establishes the API contract for distribution now so the frontend can build a "post this tweet" button against a stable interface before Twitter integration lands in P4.

#### `POST /post/batch`
- Request body: `{ "variant_ids": ["uuid-1", "uuid-2", ...] }`
- Looks up each variant in `_variants`, returns a list of `DistributionRecord`s
- Any IDs not found get skipped (or included with an error status) — partial success is fine, consistent with the graceful degradation pattern
- **Why:** The dashboard (P3) will likely want a "post all top picks" button rather than forcing users to click one at a time. This also sets up nicely for the scheduler (P5) which will want to distribute an entire pipeline run's output in one call.

#### `GET /runs`
- Returns `_runs` list (most recent first)
- Optional query param `limit: int = 10` to cap results
- Response: `list[PipelineRun]`
- **Why:** Gives the dashboard (P3) and operators a way to see pipeline history, spot failures, and monitor throughput without digging through logs.

**App configuration:**
- Title: `"AI Newsjacking Agent"`
- Prefix: none (endpoints at root)
- Entry point for running standalone: `uvicorn src.api.app:app --reload`

## 3. Tests — `tests/test_api.py` (medium)

Using `fastapi.testclient.TestClient` (sync wrapper around httpx):

| Test | What it verifies |
|------|-----------------|
| `test_get_news` | Mocks `fetch_news` → returns 2 `NewsItem`s, asserts 200 + correct list |
| `test_get_news_empty` | Mocks `fetch_news` → returns `[]`, asserts 200 + empty list |
| `test_post_run` | Mocks `run_pipeline` → returns a `PipelineRun` + 1 variant, asserts 200 + response shape |
| `test_post_run_stores_results` | Runs `/run` then `/runs`, asserts the run appears in history |
| `test_get_runs_empty` | No runs yet → asserts 200 + empty list |
| `test_get_runs_limit` | Stores 3 runs, requests `?limit=2`, asserts only 2 returned |
| `test_post_post_found` | Stores a variant via `/run`, then posts it via `/post`, asserts `DistributionRecord` with `status="pending"` |
| `test_post_post_not_found` | Posts a bogus `variant_id` → asserts 404 |
| `test_post_batch_all_found` | Stores variants via `/run`, posts all via `/post/batch`, asserts list of `DistributionRecord`s |
| `test_post_batch_partial` | Mix of valid and invalid IDs → asserts found ones get `"pending"`, missing ones get `"failed"` with error |
| `test_post_batch_empty` | Empty `variant_ids` list → asserts 200 + empty list |

All external calls (`fetch_news`, `run_pipeline`) are mocked with `unittest.mock.patch`.

## 4. Scope boundaries

**In scope (P2-T1):**
- FastAPI app with 5 endpoints
- In-memory storage for runs/variants
- Tests with mocked dependencies

**Deferred to P2-T2:**
- Background task execution (running pipeline without blocking the request)
- Detailed response models / OpenAPI schema refinement

**Deferred to P2-T3:**
- Retry middleware, structured error responses, request validation error formatting
- Health check endpoint
