## Sub-task 1: Twitter config (`src/config.py`) ‚Äî small

**Why:** The distribution module needs Twitter API credentials, and we need a way to toggle Twitter on/off so the pipeline still works in dev/test without credentials.

**What to build:**
- Add 5 new env vars to `src/config.py`:
  - `TWITTER_ENABLED` ‚Äî boolean flag, default `"false"`, controls whether posting is live
  - `TWITTER_API_KEY` ‚Äî Twitter/X app API key (consumer key)
  - `TWITTER_API_SECRET` ‚Äî Twitter/X app API secret (consumer secret)
  - `TWITTER_ACCESS_TOKEN` ‚Äî User-level OAuth 1.0a access token
  - `TWITTER_ACCESS_TOKEN_SECRET` ‚Äî User-level OAuth 1.0a access token secret
- Add `validate_twitter_config()` function:
  - Only runs when `TWITTER_ENABLED` is `true`
  - Checks all 4 credential vars are non-empty, raises `ConfigError` if any missing
  - Logs confirmation with masked key values (e.g., `TWIT...xyz`)
- Call `validate_twitter_config()` from existing `validate_config()` so it runs at API startup

**Reuses:** `ConfigError` exception class and `validate_config()` pattern from `src/config.py:16-30`

## Sub-task 2: Distribution module (`src/modules/distribution.py`) ‚Äî medium

**Why:** This is the core new module ‚Äî the actual Tweepy integration that posts content to Twitter/X.

**What to build:**
- `_get_twitter_client() -> tweepy.Client` ‚Äî lazily initializes and caches a Tweepy v2 `Client` using OAuth 1.0a user context (needed for posting tweets on behalf of a user)
- `post_tweet(variant: ContentVariant) -> DistributionRecord`:
  1. If `TWITTER_ENABLED` is `false`, return `DistributionRecord(status="pending", error="Twitter disabled")` ‚Äî allows dry-run usage
  2. Get the Tweepy client via `_get_twitter_client()`
  3. Call `client.create_tweet(text=variant.text)`
  4. Extract `response.data["id"]` as `platform_post_id`
  5. Return `DistributionRecord(variant_id=variant.id, status="posted", platform_post_id=..., posted_at=datetime.now(UTC))`
  6. On any `tweepy.TweepyException`: log the error, return `DistributionRecord(status="failed", error=str(exc))`
- Wrap `client.create_tweet()` with `@retry` from tenacity (3 attempts, exponential backoff, only retry on rate-limit / 5xx errors ‚Äî not on auth errors or duplicate tweet errors)
- Export `post_tweet` from `src/modules/__init__.py`

**Reuses:**
- `DistributionRecord` model from `src/models/distribution.py`
- `ContentVariant` model from `src/models/content.py`
- Tenacity retry pattern from `src/modules/analysis.py` and `src/modules/generation.py`
- Config vars from sub-task 1

**Character limit:** Twitter has a 280-char limit. Log a warning if `variant.text` exceeds it, but still attempt to post (let Twitter API return the error) rather than silently truncating.

## Sub-task 4: Tests (`tests/test_distribution.py`) ‚Äî medium

**Why:** Validate the module works correctly without hitting the real Twitter API.

**Test cases:**
1. **`test_post_tweet_success`** ‚Äî Mock `tweepy.Client.create_tweet` to return a fake response with `data["id"]`. Assert returned record has `status="posted"`, correct `platform_post_id`, and `posted_at` is set.
2. **`test_post_tweet_failure`** ‚Äî Mock `create_tweet` to raise `tweepy.TweepyException`. Assert returned record has `status="failed"` and `error` is populated. Verify no exception propagates (graceful degradation).
3. **`test_post_tweet_disabled`** ‚Äî Set `TWITTER_ENABLED=false`. Assert returned record has `status="pending"` without attempting any API call (mock not called).
4. **`test_validate_twitter_config_missing_keys`** ‚Äî Set `TWITTER_ENABLED=true` with missing credentials. Assert `ConfigError` is raised with a helpful message.
5. **`test_character_limit_warning`** ‚Äî Pass a variant with >280 chars. Assert a warning is logged but `create_tweet` is still called.

**Mocking approach:** Use `unittest.mock.patch` on `tweepy.Client` to avoid real API calls. Use `monkeypatch` for env vars.
