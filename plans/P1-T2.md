# P1-T2: Implement News Ingestion (CoinGecko News API)

## Overview

Implement `fetch_news()` to fetch crypto news from the CoinGecko News API, normalize responses into `NewsItem` models, extract tickers, deduplicate, and handle errors with retry/backoff.

**File:** `requirements.txt`

- Add `httpx>=0.27` — modern HTTP client, sync+async capable
- Add `tenacity>=8.0` — retry with exponential backoff (per design doc section 6.1)

## Sub-task 2: Create ingestion module (medium)

**Files:** `src/modules/__init__.py` (new), `src/modules/ingestion.py` (new)

### `fetch_news() -> list[NewsItem]`

#### 1. HTTP Call

- `GET https://api.coingecko.com/api/v3/news` using `httpx.Client`
- Wrap with `@retry(stop=stop_after_attempt(3), wait=wait_exponential(min=1, max=10))` from tenacity
- Set reasonable timeout (10s connect, 30s read)
- Set a `User-Agent` header to identify the app

#### 2. Response Mapping

CoinGecko returns an array of objects:

```
CoinGecko field     →  NewsItem field
─────────────────────────────────────
title               →  title
url                 →  url
posted_at (ISO8601) →  published_at
source_name         →  source (prefixed: "coingecko:<source_name>")
related_coin_ids    →  used for ticker extraction (see below)
```

- `content`: CoinGecko doesn't return article body text — use `title` as content (sufficient for LLM analysis in the next pipeline stage)
- Skip items where `type == "guide"` — we only want `"news"`

#### 3. Ticker Extraction

Two sources combined:

- Map `related_coin_ids` to symbols using a hardcoded lookup of top coins (e.g., `"bitcoin"→"BTC"`, `"ethereum"→"ETH"`, `"solana"→"SOL"`, ~20 entries)
- Regex scan of title for known ticker symbols as a fallback
- Deduplicate and uppercase the result

#### 4. Deduplication

Within the fetched batch:

- Normalize titles (lowercase, strip whitespace/punctuation)
- Skip items whose normalized title matches an already-seen title
- Simple exact-match after normalization (same-source dedup, sufficient for MVP)

#### 5. Error Handling (per design doc section 6.2)

- If API returns non-2xx after retries → log warning, return empty list
- If individual item fails to parse → log warning, skip it, continue with others
- Never raise exceptions to the caller

## Sub-task 3: Tests (medium)

**File:** `tests/test_ingestion.py`

Follow existing patterns from `tests/test_models.py` (pytest fixtures, standard imports). Mock `httpx.Client.get` using `unittest.mock.patch`.

### Test Cases

| Test | What it verifies |
|------|-----------------|
| `test_fetch_news_success` | Parses a mocked 3-article response into `list[NewsItem]`, verifies all fields mapped correctly |
| `test_fetch_news_filters_guides` | Items with `type="guide"` are excluded from results |
| `test_fetch_news_ticker_extraction` | `related_coin_ids=["bitcoin","ethereum"]` maps to `tickers=["BTC","ETH"]`; also tests regex fallback from title |
| `test_fetch_news_deduplication` | Two items with same title (different casing/whitespace) → only one `NewsItem` returned |
| `test_fetch_news_api_error` | Mocked 500 response → returns empty list, no exception raised |
| `test_fetch_news_empty_response` | Empty array response → returns empty list |
| `test_fetch_news_partial_parse_failure` | One malformed item in batch → skipped, others still returned |
