# P2-T2: Integrate Pipeline with API

## Overview

Make the pipeline properly work through the FastAPI layer: async execution, run-variant linking, and queryable endpoints for runs and variants.

## Sub-tasks

### 1. Restructure in-memory stores (small)

**Why:** Currently `_runs` is a flat list and `_variants` has no association to runs. We need efficient lookup by ID and run-variant linking for all other sub-tasks.

**What to build:**
- Change `_runs: list[PipelineRun]` → `_runs: dict[str, PipelineRun]` keyed by `run.id`
- Change `_variants: list[ContentVariant]` → `_variants: dict[str, list[ContentVariant]]` keyed by `run.id`
- Update `GET /runs` to return `list(reversed(_runs.values()))[:limit]`
- Update `POST /post` and `POST /post/batch` to search across all values in `_variants` dict
- Update `POST /run` to use `_runs[run.id] = run` and `_variants[run.id] = top_variants`

**File:** `src/api/app.py`

---

### 2. Background pipeline execution (medium)

**Why:** `run_pipeline()` makes multiple LLM calls and can take 30–120s. A synchronous `POST /run` will timeout or block the server. The caller should get an immediate response and poll for completion.

**What to build:**
- Import `BackgroundTasks` from FastAPI
- On `POST /run`:
  1. Create a `PipelineRun(trigger="api")` with `status="running"`
  2. Store it in `_runs` immediately
  3. Add a background function `_execute_pipeline(run_id: str)` to `BackgroundTasks`
  4. Return the run immediately (status="running", no variants yet)
- `_execute_pipeline(run_id)`:
  1. Call `run_pipeline(trigger="api")` directly
  2. On success: update `_runs[run_id]` fields (status, news_count, variants_generated, completed_at) and store variants in `_variants[run_id]`
  3. On failure: set status="failed", store error message
- Keep `pipeline.py` as the single source of truth for pipeline logic

**File:** `src/api/app.py`

---

### 3. Add `GET /runs/{run_id}` endpoint (small)

**Why:** After `POST /run` returns a run in "running" status, the caller needs to poll for completion and retrieve results.

**What to build:**
- New endpoint `GET /runs/{run_id}` → returns `RunResponse(run=..., top_variants=...)`
- Lookup `run_id` in `_runs` dict; 404 if not found
- Pull variants from `_variants.get(run_id, [])`
- Reuse existing `RunResponse` model

**File:** `src/api/app.py`

---

### 4. Add `GET /variants` endpoint (small)

**Why:** Enables the future Streamlit dashboard (P3) to list/browse all generated content without needing to know run IDs.

**What to build:**
- New endpoint `GET /variants` with optional query params `run_id: str | None = None` and `limit: int = 50`
- If `run_id` provided: return `_variants.get(run_id, [])`, 404 if run_id not in `_runs`
- If no filter: flatten all values from `_variants` dict into a single list
- Cap results with `limit`

**File:** `src/api/app.py`

---

### 5. Manual smoke test (small)

**Test steps:**
1. `uvicorn src.api.app:app --reload`
2. `curl -X POST localhost:8000/run` → expect immediate response with `status: "running"`
3. `curl localhost:8000/runs` → see the run
4. `curl localhost:8000/runs/{run_id}` → get run + variants after completion
5. `curl localhost:8000/variants` → list all variants
6. `curl localhost:8000/variants?run_id={run_id}` → filtered variants
7. `curl -X POST localhost:8000/post -d '{"variant_id": "..."}' -H 'Content-Type: application/json'` → confirm post still works
